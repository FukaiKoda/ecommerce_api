# Project 2: E-Commerce API 🛒

> **Disclaimer:** This is my 2nd learning project in backend engineering. The goal of this project was to learn and implement advanced architectural patterns (OOP, Dependency Injection, Repository Pattern), session-based authentication, Role-Based Access Control (RBAC), and relational database transactions. 
> 
> *Note: The frontend UI (the `public/` folder) was entirely generated by AI to provide a visual interface for interacting with the API. The core focus of my learning was the backend architecture.*

## 🏗 System Architecture & Flow

This project follows **Clean Code** principles and a strict **Separation of Concerns**. The architecture isolates the routing, business logic, and database interactions into distinct layers using Dependency Injection.

### Layer Breakdown
1. **Routes:** Defines API endpoints and attaches middleware (Auth, Validation, Rate Limiting).
2. **Controllers:** Extracts data from the HTTP Request (req, res), calls the appropriate Service, and returns the HTTP Response. No business logic lives here.
3. **Services:** The core business logic. It handles validations, stock checking, password verification, etc. It does not know about SQL or Prisma.
4. **Repositories:** The data access layer. This is the only place where Prisma (the ORM) is imported and used.

### High-Level Architecture Diagram

```mermaid
graph TD
    Client[Client Browser / Mobile] -->|HTTP Request| API[Express App / Router]
    
    subgraph Middleware Layer
        API --> AuthMW[Auth Middleware]
        API --> ValMW[Joi Validation]
        API --> RlMW[Rate Limiter]
    end

    subgraph Controller Layer
        AuthMW --> AuthC[Auth Controller]
        ValMW --> ProdC[Product Controller]
        ValMW --> OrderC[Order Controller]
    end

    subgraph Service Layer (Business Logic)
        AuthC --> AuthS[Auth Service]
        ProdC --> ProdS[Product Service]
        OrderC --> OrderS[Order Service]
    end

    subgraph Repository Layer (Data Access)
        AuthS --> AuthR[Auth Repository]
        ProdS --> ProdR[Product Repository]
        OrderS --> OrderR[Order Repository]
        OrderS -.-> ProdR
    end

    subgraph Infrastructure
        AuthR -.-> DB[(PostgreSQL)]
        ProdR -.-> DB
        OrderR -.-> DB
        AuthMW -.-> Redis[(Redis Session Store)]
    end
```

## 💡 Design Patterns & Clean Code

### Dependency Injection (DI)
Classes are instantiated and wired together in the `containers` folder. Instead of hardcoding `import prisma from '...'` inside the services, the Repository is injected into the Service via the constructor. This makes the system highly testable and loosely coupled.
```javascript
// Example DI Container
const orderRepository = new OrderRepository()
const productRepository = new ProductRepository()
const orderService = new OrderService(orderRepository, productRepository)
const orderController = new OrderController(orderService)
```

### Repository Pattern
Business logic (Services) is completely decoupled from the database framework (Prisma). If we ever need to switch from Prisma to TypeORM or raw SQL, we only need to rewrite the Repository layer.

### Transactional Integrity
When an order is placed, the system must create an `Order`, create multiple `OrderItem`s, and decrement the `stock` in the `Product` table. This is handled using a **Prisma Database Transaction** (`prisma.$transaction`) to ensure that if one step fails, everything rolls back safely, preventing data corruption.

### Role-Based Access Control (RBAC)
Authorization is strictly enforced. 
- **Customers** can browse products and create orders.
- **Admins** can perform full CRUD on products and view all orders globally.
This is implemented via a clean `authorize` middleware that inspects `req.session.userRole`.

## 🚀 How to Run

1. Clone the repository and install dependencies:
   ```bash
   npm install
   ```
2. Set up your `.env` file with your `DATABASE_URL`, `REDIS_URL`, and `SESSION_SECRET`.
3. Push the Prisma schema to your database:
   ```bash
   npx prisma db push
   ```
4. Start the development server:
   ```bash
   npm run dev
   ```
5. Navigate to `http://localhost:3000` to interact with the frontend UI.
